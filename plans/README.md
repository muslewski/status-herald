# Implementation Plans — status-herald (HERALD)

Generated by the improve skill on 2026-07-02 (`plan` variant — greenfield
project bootstrap, no audit). The repo does not exist yet beyond this
directory; Plan 001 creates it. Execute in the order below unless

**What this project is**: **HERALD** — *Heads-up Engine for Rendering
Adaptive Line Displays* (npm: `status-herald`, binary: `herald`). One
convention and one zero-dependency Node CLI for rendering status surfaces
across terminal hosts and agents: Claude Code `statusLine`/hooks, Grok Build
hooks + TUI-in-tmux, tmux curtain/status-right, and later zellij/kitty.
Installed globally, configured per-project. Supports agent-agnostic curtain
cards for tmux (● WORKING etc).

**Source anchors** (external contracts these plans excerpt; re-verify on
drift): token-oracle @ `ada32e9`, agentic-sage @ `cffd055`, Claude Code
statusLine docs (code.claude.com/docs/en/statusline) fetched 2026-07-02.

**Verification baseline for every plan**: `npm install` (dev deps only),
`node --test`, `npx biome check .`. Zero runtime dependencies is a hard
invariant.

## Design decisions (recorded so nobody re-litigates)

- **Language: Node ≥20 ESM, zero runtime deps** — mirrors agentic-sage (the
  operator's stated quality bar): biome, `node --test`, release-please,
  Conventional Commits, npm global install. Python (token-oracle's stack)
  was the alternative; Node won on family-convention fit and the npm
  plugin-distribution path.
- **Global install, per-project presets** — operator decision. No
  per-project installation mode; project scoping happens in one global
  config keyed by repoId (`basename-sha8`, byte-compatible with
  agentic-sage's `repo-id.mjs`).
- **Semantic roles, not colors, in presets** — themes map
  `ok/notice/warn/crit/accent/dim` per surface; gauge thresholds ported
  from token-oracle (`85/100/120`). Two color gates (piped vs TTY) ported
  from token-oracle's colors.py doctrine.
- **Config-as-message-bus interactivity** — the CLI stays stateless; menus
  and future controls write config, every surface re-reads next tick. No
  daemon, no socket in v1.
- **Naming**: project display name lives in HERALD config; tmux mirrors it
  via native `rename-session`; Claude Code's `session_name` is display-only
  (no external write API). No third naming scheme invented.
- **Fail-open everywhere on render paths** — empty output + exit 0 beats an
  error in a status bar (agentic-sage's statusline stance).
- **Distribution: npm registry only in v1** — `npm i -g status-herald` is
  the primary install (global-install decision above); `npx status-herald`
  comes free as the zero-install trial path. pip is N/A (Node project);
  Homebrew/AUR/single-binary wait for demand (Plan 011 records the
  revisit triggers).

## Findings considered and rejected

- **Fork/extend ccstatusline** (sirmalloc/ccstatusline, 30+ widgets + TUI
  config): Claude-Code-only by design; HERALD's premise is multi-surface.
  Studied as prior art instead. Same for tmux-powerline/tmux-powerkit
  (tmux-only, shell) and zjstatus (zellij-only, WASM).
- **Adopting the i3bar JSON protocol as the core wire format**: its
  block model (`full_text`/`short_text`/click events) inspired the segment
  shape, but its stdout-stream protocol fits window-manager bars, not
  command-per-tick hosts like Claude Code/tmux. Revisit as an *additional*
  render mode in Plan 010's generalization question.
- **Python implementation**: rejected for family-convention fit (see
  decisions), not on merit.
- **Per-project install mode** (like sage's project scope): operator
  explicitly rejected — "wiser to not complicate it like that."
- **Daemon/background refresher**: rejected v1 — both launch hosts poll on
  their own timers; a daemon adds failure modes for zero launch value.
  kitty may force a cached-file variant later (Plan 010).
- **Money-alert rendering now**: token-oracle has zero cost code today
  (its plan 017 is TODO); inventing cost fields would fabricate a contract.
  `oracle-alerts` ships threshold/time-to-cap alerts; money lights up when
  oracle's schema does (gate documented in Plan 006).
- **Bidirectional tmux-name sync**: deferred — needs a which-write-wins
  policy; one-directional (`herald name` → tmux) ships in Plan 008.
- **Editing token-oracle or agentic-sage from these plans**: never —
  reciprocal hints (oracle doctor mentioning herald, etc.) belong to those
  repos' own plan sets. HERALD only reads their published files
  (forecast.json, asking breadcrumbs).
